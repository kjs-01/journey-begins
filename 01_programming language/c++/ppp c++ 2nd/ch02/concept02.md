## 2.Hello, World!
### 2.1 Programs
- 프로그램: 'what to do'
- 프로그래밍: 프로그램을 작성하고 테스트하는 것.
컴퓨터는 인간(상식이 있다는 것을 가정)이 프로세스를 이해하는 과정과 다르므로 설명하는 모든 것을 일일이 정의해줘야 한다.

### 2.2 The classic first program
C++ 프로그램을 실행하면 main이란 함수에서 시작한다.
여기서 함수란 명령의 순서의 집합이고 아래 네 가지 요소로 구성되어 있다.
```cpp
// 함수를 설명하기 위한 예시 코드
#include <iostream>
int main() {
    std::cout << "Hello, World!\n";
    return 0; // 일부 OS(Unix/Linux)에서는 리턴값으로 프로그램이 정상 종료되었는지 판별한다. 0은 정상적으로 종료되었다는 뜻
}
```

- return type: `int`로 함수를 실행한 주체에게 반환하는 타입
- name: `main`으로 함수명을 의미한다.
- parameter: 매개변수로 ()안에 들어있는 내용을 의미한다. 예시 코드상에서는 비어있다.
- function body: {}로 닫혀있는 구문

### 2.3 Compilation
`Source code` -> `Compiler` -> `Object code`
- Source code
  - human readable called source code, program text
  - suffix with cpp, h
- Compiler
  - 문법적 오류가 있는지 확인
  - 사용된 모든 단어는 의미가 정의되었는지 확인
- Object code
  - Machine readable called executable, object code, machine code
  - suffix with .obj(Windows), .o(Uinx)

> 다음 한 가지는 꼭 기억해두자. `{}`로 끝나지 않는 모든 구문, 문장의 마지막에는 `;`을 적을 것.

### 2.4 Linking
`Object code` + `Object code(library)` -> `Linker` -> `Executable program`
Compile 과정에서 발생하는 에러를 compile-time errors라고 부르는 것 처럼 Linking 과정에서 발생하는 에러를 link-time이라고 부른다. 프로그램이 실행되기 전까지 발생하지 않았던 에러를 run-time errors 또는 logic errors라고 부른다.

### 2.5 Programming environments
command-line program을 사용한다면 사용자가 직접 compile과 link를 해야하지만
IDE("interactive development environment" or "intergrated development environment")를 사용한다면 **Build**를 통해 tool이 직접 compile과 link를 수행한다. 이후 Start without Debugging을 통해 프로그램을 실행할 수 있다.

### Self-Studying
1. `g++`은 단순한 컴파일러가 아닌 컴파일의 전 과정을 지휘하는 컴파일러 드라이버이다.
   1. 전처리기(cpp): `#include`나 `#define`같은 매크로를 처리한다.
   2. 컴파일러(cc1plus): 전처리가 끝난 코드를 어셈블리어로 번역한다.
   3. 어셈블러(as): 어셈블리어를 기계어(오브젝트 파일, 이진 파일)로 변환한다.
   4. 링커(ld): 생성된 오브젝프 파일과 표준 라이브러리를 연결하여 최종 실행 파일을 만든다.
   
   즉, 링커는 존재하지만, `g++`이 알아서 마지막 단계에서 링커를 호출해주기 때문에 편의상 `g++`을 사용해 라이브러리를 연결한다고 표현한 것이다. 링커를 실행하지 않고 순수하게 오브젝트 파일까지만 생성하고 싶다면 `-c` 옵션을 사용하면 된다. 예시는 다음과 같다. `g++ -c hello_world.cpp`

    **전처리기의 역할**
    - 코드 복사 및 붙여넣기: `#include <iostream>`이라는 줄을 만나면 컴퓨터 내부에 저장된 실제 `iostream` 헤더 파일을 열어서 그 안에 있는 수천 줄의 코드를 통째로 복사한 뒤 파일 상단에 붙여넣는다.
    - 단어 치환: `#define MAX 100`이라고 적혀 있다면, 코드 내에 있는 모든 `MAX`라는 글자를 찾아 `100`으로 치환한다.

    **헤더 파일과 컴파일러의 역할**

    - 헤더 파일: 컴파일러의 에러를 방지하기 위한 코드의 선언만 존재함.
      - 선언(Declaration): 중괄호 없이 세미콜론(;)으로 끝난다.<br>
        `int add(int a, int b);`
      - 정의(Definition): {}중괄호 안에 구체적인 함수가 구현되어 있다.
        ```cpp
          int add(int a, int b) {
          return a+b;
        }
        ```
    - 컴파일러: 헤더 파일에 선언된 코드를 보고 에러 없이 오브젝트 코드로 번역한다. 실제 오브젝트 파일 안에는 `cout`의 코드가 없으므로 "나중에 `cout` 진짜 주소를 여기다 채워 넣어라"하고 빈 칸을 뚫어 놓는다.

    **링커와 표준 라이브러리의 역할**

    - 표준 라이브러리: `cout`의 진짜 동작 코드(정의)는 완성된 기계어 형태로 '표준 라이브러리'라는 거대한 파일에 별도로 저장되어 있다.
    - 링커: 표준 라이브러리 파일을 뒤져서 `cout`의 진짜 기계어 코드를 찾아낸 다음 오브젝트 파일에 뚫려 있던 빈칸에 코드를 연결(Link)해 준다. 이렇게 해서 최종 실행 파일이 완성된다.
  
    **오브젝트 파일(선언/정의, 컴파일/어셈블)에 대한 이해**

    - **심볼 테이블(Symbol Table):** 현재 코드에서 만들어낸 함수/변수의 이름과, 외부에서 빌려와야 하는(아직 주소를 모르는) 이름들의 목록을 적어두는 장부이다.
    - **재배치 테이블(Relocation Table):** 선언만 되어있어 나중에 채워줘 라는'Flag'가 존재하며 "코드의 어느 위치에 있는 빈칸을, 어떤 심볼의 진짜 주소로 덮어씌워야 하는지"에 대한 작업 지시서이다.

      심볼 테이블과 재배치 테이블을 활용한 컴파일/어셈블 과정은 아래와 같다.

      1. **어셈블러의 장부 작성** 
         - 내부 심볼: `main`이나 내가 직접 만든 변수는 진짜 주소를 계산하여 심볼 테이블에 적어둔다.(상태: Defined)
         - 외부 심볼: `cout`이나 `keep_window_open()`같은 함수를 만난다. 당장 진짜 주소를 모르기 때문에 기계어 코드의 해당 위치에 에러를 내는 대신 일단 `0x0000000`같은 가짜 주소(Dummy Address)를 채워 넣는다.(상태: Undefined)
      2. **Flag 남기기:** 어셈블러가 가짜 주소를 넣고 재배치 테이블에 Flag를 남길 때, 심볼 테이블의 인덱스 번호를 적어둔다.
   
         > 링커야, 코드 시작점으로부터 120바이트 떨어진 곳에 빈칸이 하나 있어. 나중에 심볼 테이블의 5번 항목(`cout`)의 진짜 주소를 찾게 되면, 그 주소를 가져와서 이 빈칸에 덮어씌워 줘!"

      3. **링커의 심볼 해석 및 빈칸 채우기(링킹 과정)**
         1. 장부 통합: 링커는 여러 개의 오브젝트 파일(`.o`)과 표준 라이브러리를 끌어모은 뒤, 각 파일이 가지고 있던 심볼 테이블들을 하나로 합쳐 거대한 '전역 심볼 테이블'을 만든다.
         2. 짝 맞추기: 통합된 장부를 훑어보면서 Undefined 상태였던 심볼(예: 내 파일의 `cout`)과, 라이브러리에 있던 Defined 상태로 있던 심볼(표준 라이브러리의 `cout` 진짜 주소)의 짝을 찾아 연결한다.
         3. 빈칸 채우기: 장부가 완성되면 링커는 비로소 재배치 테이블(Flag)을 확인한다. "아, 120바이트 위치에 심볼 테이블 5번을 넣으라고 했지?" 방금 완성된 심볼 테이블에서 5번(`cout`)의 진짜 주소를 읽어롸 빈칸에 쏙 채워 넣는다.
   
    **링커가 짝을 찾는 진짜 기준**
    1. 재배치 테이블의 지시: "코드 120바이트 위치에, 내 심볼 테이블의 5번 항복 진짜 주소를 넣어라."
    2. 심볼 테이블 확인: 링커가 해당 오브젝트 파일의 심볼 테이블 5번 항목을 열어본다. 이 안에는 다음과 같은 정보가 들어있다.
       - 상태: Undefined
       - 이름: `"cout"`(정확한 문자열 이름이 저장되어 있음)
    3. 글로벌 탐색 (문자열 매칭): 이제 링커는 인덱스 번호를 버리고, 알아낸 이름인 `"cout"`이라는 문자열 자체를 들고 표준 라이브러리와 다른 파일들의 심볼 테이블을 탐색한다.
    4. 매칭 및 주소 복사: 표준 라이브러리의 심볼 테이블에서 이름이 `"cout"`이고 상대타 Defined인 항목을 발견한다. 그러면 그곳에 적힌 '진짜 메모리 주소'를 복사해 와서, 처음 지시 받았떤 120바이트 위치의 빈칸에 덮어씌운다.

    **네임 맨글링(Name Mangling)**

    위의 개념들과 이어서 C++에는 '함수 오버로딩(Overloading)'기능이 있다. 
    ```cpp
    int add(int a, int b); // 1번 add
    double add(double a, double b); // 2번 add
    ```
    만약 링커가 단순히 `"add"`라는 이름표만 들고 짝을 찾는 다면 1번을 써야할 지 2번을 써야할 지 알 수 없다.

    그래서 C++ 컴파일러는 심볼 테이블에 이름을 적을 때, 사람이 원래 쓴 이름(`add`)를 그대로 적지 않고 매개변수의 타입 등의 정보르 섞어서 암호 같은 문자열로 이름을 변형해서 저장한다. 이를 네임 맨글링(Name Mangling)이라고 한다.

    - int add(int, int) -> 심볼 테이블에는 `_Z3addii`로 저장됨 (i는 int를 의미)
    - double add(double, double) -> 심볼 테이블에는 `_Z3adddd`로 저장됨 (d는 double을 의미)
    
    따라서 링커는 단순히 `add`를 찾는 것이 아니라, 컴파일러가 변형해 둔 `"_Z3addii"`라는 고유하고 정확한 문자열을 기준으로 글로벌 심볼 테이블을 탐색하기 때문에 오차 없이 정확한 함수의 정의를 찾아서 연결할 수 있다.

2. (Drill-3.컴파일 과정)을 요약해 보겠다.
   1. 컴파일러의 역할: 정보 파악 및 '작업 지시서' 작성
   
      컴파일러는 C++ 코드를 어셈블리어(`.s`파일)로 번역하면서 코드의 문맥을 모두 파악한다.

      이때 컴파일러는 코드에 등장하는 모든 변수와 함수의 상태를(Defined인지, Undefined인지)를 분류하고 앞서 언급한 네임 맨글링(Name Mangling)을 수행하여 고유한 이름을 만들어 낸다.

      하지만 컴파일러는 직접 오브젝트 파일(`.o`)를 조작하지 않기 때문에, 자신이 파악한 정보를 다음 단계인 어셈블러에게 전달하기 위해 어셈블리어 코드 사이에 '어셈블러 지시어(Directive)'라는 특별한 메모를 남겨둔다.

     - **다음은 실제 컴파일러가 만들어낸 어셈블리어의 예시(개념적):**
       ```assembly
       .extern _Z4cout  ; 메모: "어셈블러야, '_Z4cout'라는 애는 외부에 있는(Undefined)야!"
       .globl main      ; 메모: "main 함수는 여기서 정의된(Defined) 애니까 외부에 공개해 줘!"
   
       main:            ; 진짜 main 함수의 어셈블리 코드 시작
          ...
          call _Z4cout  ; 메모: "여기서 아까 말한 그 함수 호출해!"
       ```

   2. 어셈블러의 역할: 진짜 '테이블' 생성 및 기록
      
      어셈블러는 컴파일러가 넘겨준 이 어셈블리어 파일(`.s`)를 한 줄 한 줄 읽으면서 0 과 1의 기계어(오브젝트 파일, `.o`)로 변환한다.

      이때 컴파일러가 남겨둔 지시어(`.extern`, `.global` 등)을 읽고, 비로소 오브젝트 파일 내부에 물리적인 공간을 할당하여 심볼 테이블과 재배치 테이블을 작성한다.

      1. 심볼 테이블 작성: 어셈블러가 `.extern _Z4cout`를 읽는다. 
         > "아, 컴파일러가 이 녀석은 외부에 있다고 했지?" 하면서 심볼 테이블에 이름은 `_Z4cout`, 상태는 Undefined, 주소는 비어있음으로 기록한다.
      
      2. 재배치 테이블 작성: 어셈블러가 `call _Z4cout`를 기계어로 바꾸려고 한다.
         > "당장 주소를 모르니까 기계어에는 0x00(가짜 주소)을 넣어야겠다. 그리고 링커를 위해 재배치 테이블에 '여기 방금 가짜 주소 넣었으니 나중에 심볼 테이블에서 `_Z4cout` 찾아서 채워달라'고 Flag를 꽂아둬야지!"
         
3. C++ 언어는 과거 메모리 부족의 한계로 (Single-pass Compiler) "컴파일러가 읽기 전에 미리 위에서 선언해 줘!" 라는 규칙이 생겼고 헤더 파일과 소스 파일이 분리되는 생태계의 원인이 되었다.
   
   (Java, C#, Python 같은 언어는 메모리가 충분히 커진 시대에 나와서 코드를 여러번 훑기 때문에 함수의 순서가 상관없다.)

4. 어셈블리어가 오늘날에도 존재하는 세 가지 이유
   1. 기계어는 인간이 절대 '해독'할 수 없다.
      
      인간의 눈으로 기계의 동작을 디버깅하기 위해 존재하는 소프트웨어의 현미경

   2. 하드웨어(CPU)를 통제하기 위한 유일한 언어
      
      C나 C++은 훌륭한 언어지만, '범용적'으로 만들어져서 컴퓨터의 특정 하드웨어를 직접적으로 건드리는 기능은 언어 문법 자체에 존재하지 않는다.

      특히 운영체제의 기초를 설계하고 학습하는 과정에 진입하면 이러한 고수준 언어로는 접근이 불가능하다.

   3. 컴파일러 제작의 효율성 (플랫폼 독립성)

      Intel, AMD, ARM 등 다양한 종유의 CPU가 존재한다.

      만약 컴파일러가 C++을 곧바로 기계어로 번역하면 C++ 컴파일러를 만드는 사람은 Intel용, AMD용, ARM용 컴파일러를 바닥부터 완전히 따로 만들어야 한다.

      중간에 어셈블리어라는 단계가 존재하면 일이 쉬워진다.

         1. 공통 번역(컴파일러): C++ 코드를 분석해서 
         2. 맞춤형 조립(어셈블러): 각 하드웨어에 맞게 어셈블러-기계어로 1대1 변환
   
   결론이다. 프로그래밍 언어 N개, 아키텍쳐가 M개가 있다고 가정하자.
      
   공통된 어셈블리어인 LLVM IR(Low-Level Virtual Machine Intermediate Representation)로 변환해서 각 아키텍쳐용 기계어로 조립하면 N*M의 경우의 수가 N+M으로 줄어들기 때문에 유용하다.

5. g++ 컴파일 단계별 옵션
   
   |옵션|진행되는 단계|결과물 확장자|파일의 정체|
   |---|---|----|---|
   |-E|전처리 단계만 실행|`.ii`(또는 화면 출력)|소스 코드에 헤더 파일이 통째로 복사된 텍스트 파일|
   |-S|전처리 + 컴파일까지 실행|`.s`|어셈블리어 코드 파일|
   |-c|전처리 + 컴파일 + 어셈블까지 실행|`.o`|기계어(이진) 오브젝트 파일|
   |없음(또는 `-o`)|-c + 링킹까지 전체 실행|확장자 없음 (윈도우는`.exe`)|실행 가능한 완성된 바이너리 파일|

   `-c`(compile), `-o`(output)는 소문자이고 `-E`(Preprocess Expansion 혹은 Edit), `-S`(Source)는 대문자인 이유는 아래와 같다.

   Unix 시스템 명령어 자원 선점 경쟁이다. c o 옵션은 많이 사용했고, s e 는 strip(디버깅 심볼 테이블 삭제 -> 실행 파일 용량 줄이기), entry point가 먼저 선점하게 됐다.

6. 그래서 실행되는 파일 이름을 main으로 하는 거겠구만. 소스파일(cpp)은 함수의 정의(구현), 헤더 파일은 소스 파일의 선언. main함수에서는 구현된 함수의 조합. 이런 느낌이지 않을까~ 하고 생각해봅니다.
