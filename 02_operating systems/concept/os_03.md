## 03_Process(Limited Direct Execution)
운영체제는 물리적 CPU를 time sharing을 통해서 효과적으로 가상화를 진행한다.
성능적인 부분: 시스템에 overhead를 발생시키지 않고 가상화를 실행하는 방법은 없을까?
제어권: CPU로 부터 제어권을 얻는 동안 어떻게 효율적으로 프로세스를 동작할 수 있을까?

### Direct Execution

| OS                               | Program                       |
| -------------------------------- | ----------------------------- |
| 1. Create entry for process list |                               |
| 2. Allocate memory for program   |                               |
| 3. Load program into memory      |                               |
| 4. Set up stack with argc / argv |                               |
| 5. Clear registers               |                               |
| 6. Execute call main()           |                               |
|                                  | 7. Run main()                 |
|                                  | 8. Execute return from main() |
| 9. Free memory of process        |                               |
| 10. Remove from process list     |                               |
|                                  |                               
--> OS선에서 커널, 메모리, 디스크, I/O다 건들일 수 있는 경우.

> 동작하는 프로그램에 아무런 제어가 없다면 OS는 아무 역할을 하지 않는다.<br>
> 위와 같은 Direct Execution은 제한된 동작을 하는 문제가 생긴다.
> 1. 디스크에 I/O request를 발생시킬 때.
> 2. CPU 나 Memory같은 시스템 리소스를 더 많이 할당 받으려 할 때.
>
> 해결 방법은 아래와 같이 두 가지의 모드로 나눈다.
> 1. User Mode: 어플리케이션은 하드웨어 리소스에 모두 접근할 수는 없다.
> 2. Kernel Mode: 운영체제는 기계에 대한 모든 리소스 접근이 가능하다.

### System Call
커널이 특정 핵심 기능(하드웨어 수준의)들을 사용자 프로그램에 신중하게 노출시키기 위해 발생시킨 것.
System Call의 예시는 아래와 같다.
- 파일 시스템에 접근
- 프로세스의 생성과 삭제
- 다른 프로세스들과의 연관
- 메모리 추가 할당
- Trap: kernel 모드로 권한 격상
- Return-from-trap: 호출한 유저 프로그램으로 되돌아 간다. user mode로 권한 격하

어떻게 trap이 OS 내부에서 동작해야 할 코드가 어떤건지 알아?
- trap table
- trap handler

System-call number : 각 시스템 콜에 할당된다.
사용자 코드는 원하는 시스템 콜 넘버를 레지스터에 할당해야 한다.

### Limitied Direction Execution Protocol

| OS@ boot(kernel mode) | Hardware                | Program(user mode) |
| --------------------- | ----------------------- | ------------------ |
| initialize trap table |                         |                    |
|                       | remember address of ... |                    |
|                       | syscall handler         |                    |

| OS@ run(kernel mode)          | Hardware                       | Program(user mode) |
| ----------------------------- | ------------------------------ | ------------------ |
| Create entry for process list |                                |                    |
| Allocate memory for program   |                                |                    |
| Load program into memory      |                                |                    |
| Setup user stack with argv    |                                |                    |
| Fill kernel stack with reg/PC |                                |                    |
| **return-from-trap**          |                                |                    |
|                               | restore regs from kernel stack |                    |
|                               | move to user mode              |                    |
|                               | jump to main                   |                    |
|                               |                                | Run main()         |
|                               |                                | ...                |
|                               |                                | Call system        |
|                               |                                | trap into **OS**   |

핵심은 아래와 같다. 
**kernel -> user 인 경우는**<br>
restore regs from kernel stack
jump to PC/main() etc..
**user -> kernel 인 경우는**<br>
save regs to kernel stack
jump to trap

1. 프로세스 리스트의 진입점을 생성
2. 프로그램을 위한 메모리 할당
3. 프로그램을 메모리로 올림
4. 유저 스택에 argv 셋업
5. 커널 스택에 reg/PC 채움
6. kernel mode -> user mode(return-from-trap)
> 프로세스마다 PCB, 가상 메모리, kernel stack 이 따로 존재한다.

이래도 문제가 생깁니다. 프로세스 사이에 교체를 할 때죠. 그러면 OS는 CPU로 부터 어떻게 제어 권한을 얻을 수 있을까?
- 협조적인 방식: 시스템 콜을 기다린다
  - CPU 제어권을 놓는다 yield를 통해서.
- 비협조적인 방식: OS가 강제로 제어권을 가져온다 -> 필요 이상의 시간이 소비되면

### Timer Interrupt(비협조적인 방식)
1. OS가 timer interrupt를 실행
2. 하드웨어가 시스템 콜 핸들러와 타이머 핸들러의 주소를 떠올려
3. OS가 타이머 인터럽트를 시작해
4. 하드웨어가 타이머를 시작하고 X ms 후에 CPU에 인터럽트 신호를 보내
5. 유저 모드(프로그램)이 프로세스 A를 실행하고 있어
6. 하드웨어에서 타이머 인터럽트가 발생하면 프로세스 A의 정보를 old 커널 스택에 저장해.
> 각 프로세스마다 커널 스택이 따로 존재한다고 했으니까 각 레지스터가 요구하는 값들을 주소에 맞춰서 저장했다는 뜻이곘지.
7. 트랩을 발동해(커널 모드로 넘어가)서 스위치 루틴을 실행하지
   1. 여기서 A 프로세스의 정보를 old 커널 스택에 저장하고 현재 프로세스 구조체에 B 프로세스의 정보가 담겨있는 커널 스택으로 교체한다.
8. 교체된 커널 스택에서 B 프로세스의 데이터를 복구하고 유저모드로 return-from-trap한다.

### Scheduler
결정을 내린다. 현재 프로세스를 지속해서 동작할건지, 아니면 다른 프로세스로 변경할건지
만약 변경하는 걸로 결정이 나면 OS가 Context Switch를 실행

### Context Switch
저레벨의 어셈블리 코드
- 커널 스택에 현재 프로세스에 대한 몇 개의 레지스터 값을 저장한다.
  - 일반 목적의 레지스터
  - PC (Program Counter)
  - 커널 스택 포인터
- 곧 실행될 프로세스의 커널 스택에서 몇 개 값을 복구한다.
- 곧 실행될 프로세스의 커널 스택으로 변경한다.
요약하자면 현재 프로세스 값들을 커널 스택에 저장하고, 변경할 프로세스의 커널 스택에서 몇 가지 정보를 가져 온 후에 교체한다.

## 추가 정보 
### 1. 문맥 교환 (Context Switch)

* **정의:** CPU가 바라보는 **스택 포인터(`%esp`)를 프로세스 A에서 B로 바꿔치기**하는 작업입니다.
* **원리:** `%esp` 주소만 바꾸면 CPU의 시선이 B의 스택으로 넘어갑니다. 이후 스택에서 값을 빼내어(`pop`) B의 레지스터 상태와 복귀 주소(`%eip`)를 복원합니다.

### 2. 스택 방향과 'Top'의 역설

* **방향:** 스택 메모리는 **높은 주소에서 낮은 주소로** 자라납니다. (`push` 하면 주소가 작아짐)
* **Top의 의미:** 따라서 현재 스택의 꼭대기(**Top, `%esp**`)는 물리적으로 **주소값이 가장 작은 곳**입니다.

### 3. C 언어 포인터와의 절묘한 조화

* **규칙:** C 언어의 구조체/배열 포인터는 무조건 **가장 낮은 주소**를 가리킵니다.
* **매핑:** OS가 레지스터를 스택에 밀어 넣으면 최종 `%esp`(Top)가 가장 낮은 주소에 멈춥니다. 이 주소를 C 언어 포인터(`context`)로 넘기면 오차 없이 메모리가 1:1로 연결됩니다.

### 4. 첫 프로세스 실행 기법 (Stack Faking)

* 멈춘 적이 없는 첫 프로세스는 스택에 **가짜 레지스터 값과 첫 실행 함수 주소**를 몰래 써넣어 둡니다.
* CPU는 원래 실행되다가 멈췄던 프로세스로 착각하고 자연스럽게 `pop`을 하며 실행을 시작합니다.

### 5. 핵심 레지스터 역할 분담

* **`%esp` (스택 포인터):** 프로세스의 데이터와 상태가 저장된 **스택의 꼭대기 주소**를 기억합니다.
* **`%eip` (명령어 포인터):** CPU가 다음에 실행할 **코드의 주소**를 기억합니다.

### 6. 지역 배열의 스택 할당 방식

* 스택 기준선(`%esp`)이 흔들리면 위험하므로 배열에 값을 넣을 때 `push`나 `pop`을 쓰지 않습니다.
* `sub` 명령어로 **공간을 한 번에 확보**하여 `%esp`를 고정한 뒤, `mov` 명령어와 거리 계산(오프셋)을 통해 정해진 위치에 값을 직접 꽂아 넣습니다.
