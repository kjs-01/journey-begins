## 02_Process(intro)
프로세스는 실행 중인 프로그램이다.<br>
프로세스는 실행되고 있는 프로그램의 인스턴스(사례)이다.<br>
프로세스는 자원 할당과 스케줄링의 단위이다.

프로세스는 운영체제로 부터 독립된 가상의 메모리 공간을 부여받아 하나 이상의 스레드를 담고 실제로
작업을 수행하는 프로그램의 실행 주체이다.

운영체제는 프로세스를 관리하기 위해 다음 세 가지 정보를 가지고 있다.
- 각 프로세스가 어디까지 실행됐는지(Program Counter 값)
- 어떤 메모리 지도를 쓰고 있는지(Page Table 주소)
- 프로세스의 식별자가 무엇인지(PID)
이 모든 정보를 PCB(Process Control Block)이라는 구조체에 담아서 관리

### 프로그램을 실행한 이후
운영체제는 페이지 테이블을 렘의 커널 영역에 생성한다.

### PTE (Page Table Entry)
| 가상 페이지 번호 (구역)| Valid Bit (존재 여부)| 실제 데이터가 있는 위치 (핵심!) |
| --- | --- | --- |
| 0번 (Code 영역 시작)| 0 (Invalid)| 디스크 블록 500번지 (실행 파일 위치) |
| 1번 (Data 영역) |0 (Invalid)| 디스크 블록 502번지 (실행 파일 위치) |
| ... | ... |... |
| 99번 (Heap 영역) | 0 (Invalid) | 없음 (아직 안 태어남) |
| 100번 (Stack 영역) | 0 (Invalid) | 없음 (아직 안 태어남) |
---

> PCB (Process Control Block) 에는 PID, PC 레지스터 값, 페이지 테이블 기준 주소가 저장된 구조체로 PTE와 다르다. PID를 부여해야 진짜 프로세스가 생성되는 거라고 생각하면 됨.
> PCB 구조체를 먼저 생성하고 다른 가상 메모리 주소에 PT(Page Tabel의 한줄 한줄을 PTE라고 부름)를 생성한다.

1. CPU의 지시: "MMU야, 가상 0번 페이지 가져와!"
2. MMU의 확인: MMU가 RAM에 있는 페이지 테이블을 재빨리 뒤져봅니다.
3. 함정 발동(Trap): "어? Valid Bit가 0이네? 물리 RAM에 없잖아!" MMU는 데이터를 가져오는 대신, CPU에 강제로 전기 신호를 보내 작업을 중단시킵니다. 이것이 바로 **페이지 폴트(Page Fault)** 라는 이름의 하드웨어 예외(Exception)입니다.

> Page Fault : 물리 렘에 실제 데이터가 없다. CPU 잠깐 멈춰! 라고 하는 예외다. MMU(Memory Management Unit)가 발생시킨다.

CPU가 작업을 멈추면 **Context Switch**(OS가 잠에서 깨어나 상황 통제)가 발생한다.

1. 원인 파악: OS가 장부(PTE)를 봅니다. "아하, 가상 0번 페이지 코드를 읽으려다 걸렸구나. 이 녀석 원래 주소는 디스크 블록 500번지네."
2. 빈방 찾기: OS는 물리 메모리(RAM)를 훑어보고 텅 빈 공간(예: 물리 프레임 12번)을 하나 찾습니다.
3. 디스크에 명령: OS는 **하드디스크 컨트롤러**에게 명령합니다. "디스크 500번지 블록을 물리 RAM 12번 프레임으로 당장 복사해!"
4. 프로세스 수면: 디스크가 돌아가는 속도는 CPU 입장에서 억겁의 시간입니다. OS는 이 프로세스를 **'대기(Blocked) 상태'**로 잠재우고, 다른 프로세스에게 CPU를 넘겨줍니다.
5. DMA 복사: 디스크 장치와 RAM은 CPU를 괴롭히지 않고 둘이서 알아서(DMA, Direct Memory Access) 데이터를 복사합니다. 다 끝나면 디스크가 OS에게 "복사 끝!"이라고 알림(Interrupt)을 보냅니다.

### Process_Creation
fork() : 부모 프로세스의 정보를 가지고 자식 프로세스를 생성
제일 중요한 **fork()를 하는 이유**, 프로세스를 생성할 때마다 권한을 부여하고, 커널 리소르를 연결하는 과정이 비효율적이니까 기존에 존재하는 부모 프로세스의 상태(권한, 연결된 I/O etc)를 복사해서 원하는 작업을 수행하자(exec())는 취지. 하지만 결국 프로세스는 다 실행되어야 하므로 PID를 통해 관리할 수 있도록.
1. PCB 구조체를 생성하고 초기화한다.
2. 새로운 주소 공간을 생성하고 초기화한다.
3. 생성한 주소 공간에 부모의 주소 공간의 전체 복사본을 채워 넣는다.
4. 부모가 사용하던 커널 리소스를 가리키도록 설정한다.
5. PCB를 레디 큐(스케줄러가 관리하는 대기열)에 배치한다.
6. 부모에게는 자식 프로세스의 PID를 리턴하고, 자식 프로세스는 0을 리턴한다.

exec() : 부모 프로세스의 위치에 자식 프로세스가 실행한 정보를 덮어씀
1. 현재 프로세스를 멈춘다.
2. 자식 프로세스 주소 공간에 프로그램 "prog"를 로드한다.
3. 새로운 프로그램을 위해 하드웨어 context(상태)와 args(인자)를 설정한다.
4. PCB를 레디 큐에 배치한다.
5. exec()는 새로운 프로세스를 생성하지 않는다.
6. exec()이 리턴된다는 건 무슨 뜻일까요?
> exec()이 정상적으로 실행될 경우, 원래 프로그램 코드는 새 프로그램으로 완전히 덮어씌워져 사라진다. 원래 코드가 지워졌으니, exec()함수 밑에 적어둔 코드는 실행되지 않으며 return 할 곳 자체가 없다.
> 하지만, 반환이 된다면 이는 Error라는 뜻이다.(예를 들어, 덮어씌울 "prog"파일이 존재하지 않거나, 권한이 없는 경우)

### Data Structure
- 프로세스 리스트
- 레지스터 컨텍스트
- PCB

### PCB
- process state: Running, Ready, Waiting(Blocked)
- program counter: CPU가 다음번에 읽어들일 명령어의 주소
- cpu register: 프로세스가 멈추고 다른 프로세스로 교체(context switch)될 때, CPU 내부 레지스터에 저장되어 있던 임시 계산값들을 통째로 백업해 두는 공간
- cpu scheduling information: 프로세스의 중요도, 대기열에서의 위치 정보
- memory management information: 프로세스가 사용 중인 메모리 공간에 대한 설명
- process accounting infromation: 프로세스가 태어난 지 얼마나 지났는지, CPU를 몇 초나 썻는지 기록(너무 혼자 오래 쓰면 강제로 쫓아내기 위함)
- I/O status information: 이 프로세스가 어떤 입출력 장치를 잡고 있는지, 어떤 파일을 열어두고 있는지 기록
