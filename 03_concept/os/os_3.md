## 03_Process(Limited Direct Execution)
운영체제는 물리적 CPU를 time sharing을 통해서 효과적으로 가상화를 진행한다.

1. 프로세스 리스트의 진입점을 생성
2. 프로그램을 위한 메모리 할당
3. 프로그램을 메모리로 올림
4. 유저 스택에 argv 셋업
5. 커널 스택에 reg/PC 채움
6. kernel mode -> user mode(return-from-trap)
> 프로세스마다 PCB, 가상 메모리, kernel stack 이 따로 존재한다.

그러면 OS는 CPU로 부터 어떻게 제어 권한을 얻을 수 있을까?
- 협조적인 방식: 시스템 콜을 기다린다
- 비협조적인 방식: OS가 강제로 제어권을 가져온다 -> 필요 이상의 시간이 소비되면

### Scheduler
결정을 내린다. 현재 프로세스를 지속해서 동작할건지, 아니면 다른 프로세스로 변경할건지
만약 변경하는 걸로 결정이 나면 OS가 Context Switch를 실행

### Context Switch
저레벨의 어셈블리 코드
- 커널 스택에 현재 프로세스에 대한 몇 개의 레지스터 값을 저장한다.
  - 일반 목적의 레지스터
  - PC (Program Counter)
  - 커널 스택 포인터
- 곧 실행될 프로세스의 커널 스택에서 몇 개를 되돌린다.
- 곧 실행될 프로세스의 커널 스택으로 변경한다.
요약하자면 현재 프로세스 값들을 커널 스택에 저장하고, 변경할 프로세스의 커널 스택에서 몇 가지 정보를 가져 온 후에 교체한다.

### Timer Interrupt
1. OS가 timer interrupt 를 실행해
2. 하드웨어가 시스템 콜 핸들러와 타이머 핸들러의 주소를 떠올려
3. OS가 타이머 인터럽트를 시작해
4. 하드웨어가 타이머를 시작하고 X ms 후에 CPU에 인터럽트 신호를 보내
5. 유저 모드(프로그램)이 프로세스 A를 실행하고 있어
6. 하드웨어에서 타이머 인터럽트가 발생하면 프로세스 A의 정보를 old 커널 스택에 저장해.
> 각 프로세스마다 커널 스택이 따로 존재한다고 했으니까 각 레지스터가 요구하는 값들을 주소에 맞춰서 저장했다는 뜻이곘지.
7. 트랩을 발동해(커널 모드로 넘어가)서 스위치 루틴을 실행하지
   1. 여기서 A 프로세스의 정보를 old 커널 스택에 저장하고 현재 프로세스 구조체에 B 프로세스의 정보가 담겨있는 커널 스택으로 교체한다.
8. 교체된 커널 스택에서 B 프로세스의 데이터를 복구하고 유저모드로 return-from-trap한다.
